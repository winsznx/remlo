# Core REST API

This is the first-party API used by the Remlo web app and operator surfaces. Use it when you are acting as an authenticated user inside the product. Use [MPP Endpoints](mpp-endpoints) when a machine client should pay per request or session.

## Authentication

Most routes expect:

```http
Authorization: Bearer <privy-jwt>
```

The server decodes the Privy token, resolves the caller’s role, and then authorizes the requested employer, employee, or admin action.

## Route groups

| Method | Route | Purpose |
| --- | --- | --- |

| `POST` | `/api/employers/[id]/team` | Create an employee under an employer |
| `PATCH` | `/api/employers/[id]/team/[employeeId]` | Update salary, pause or resume payments, remove an employee, or record review actions |
| `GET` | `/api/employers/[id]/treasury` | Return treasury balances and related treasury data |
| `GET` | `/api/employers/[id]/compliance` | Employer compliance summary and audit log |
| `GET` | `/api/employers/[id]/mpp-sessions` | Recent MPP sessions |
| `GET` | `/api/employers/[id]/mpp-receipts` | Aggregated MPP receipt activity |
| `POST` | `/api/employers/[id]/agent-key` | Generate or rotate the one-time agent key |

## Important route behavior

### Employer creation

`POST /api/employers` is idempotent for the same Privy owner. If the user already owns a workspace, the existing employer id is returned.

### Payroll preparation

`POST /api/employers/[id]/payroll` is the pre-flight route behind the payroll wizard. It:

1. parses payroll items
2. converts amounts into six-decimal token units
3. checks treasury availability
4. runs TIP-403 authorization checks when required
5. builds memo bytes
6. writes draft payroll state into Supabase
7. returns unsigned calldata for `executeBatchPayroll`

### Employee creation and invite reuse

`POST /api/employees` now distinguishes between:

- a new employee record
- an existing but still claimable employee record
- an already claimed employee record

That keeps the UI from pretending an already-claimed invite can still be reused.

### Team payment holds

`PATCH /api/employers/[id]/team/[employeeId]` now treats payment holds as a real state transition:

- pausing an already paused employee returns `409`
- resuming an already active employee returns `409`

## Common query parameters

### `GET /api/transactions`

| Parameter | Type | Notes |
| --- | --- | --- |
| `page` | integer | 1-indexed, default `1` |
| `limit` | integer | Max `100`, default `25` |
| `employeeId` | string | Filter by employee id |
| `status` | string | Filter by payment status |
| `from` | ISO timestamp | Lower bound for `created_at` |
| `to` | ISO timestamp | Upper bound for `created_at` |
