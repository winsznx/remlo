# Auth and Access

Remlo separates **identity** from **role**.

- Privy handles sign-in and wallet creation
- Supabase determines whether that identity is an employer, employee, or admin
- Middleware routes the user into the right workspace

## Supported sign-in methods

- Email
- SMS
- External wallet connection
- Passkey from the dedicated login action

That combination keeps onboarding low-friction for non-crypto users while still supporting wallet-native operators.

## Who lands where

| User state | Destination |
| --- | --- |
| Unauthenticated visitor hitting a protected route | `/login` |
| Employer | `/dashboard` |
| Employee | `/portal` |
| Platform admin allowlisted through `ADMIN_USER_IDS` | `/admin` |
| Authenticated user with no resolved role | `/register` |

## The role model

### Employer

An employer is resolved from the Privy user DID that owns the employer record. This user operates treasury, team management, payroll, compliance, and API access.

### Employee

An employee claims an invite, links that identity to the employee record, and then uses `/portal`.

### Admin

Admins are a separate internal allowlist and should not be treated as customer-facing product roles.

## Public versus protected routes

### Public routes

- `/`
- `/pricing`
- `/docs`
- `/about`
- `/contact`
- `/login`
- `/register`
- `/invite/[token]`
- `/kyc/[token]`
- `/legal/*`
- webhook routes

### Protected route families

| Prefix | Expected role |
| --- | --- |
| `/dashboard` | Employer |
| `/portal` | Employee |
| `/admin` | Platform admin |

## API authentication

The first-party REST API expects:

```http
Authorization: Bearer <privy-jwt>
```

Server helpers decode the Privy token, resolve the caller’s role, and then authorize employer, employee, or admin actions.

## Testing guidance

- Use one identity for the employer workspace
- Use a separate email, phone number, or wallet for employee invite testing
- Do not reuse the employer identity when claiming an employee invite, or you will keep landing in `/dashboard`
