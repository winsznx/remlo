# Memo and Compliance Model

Every serious payroll movement in Remlo carries two layers of context:

1. a payroll memo that links the transfer back to business context
2. a TIP-403 policy check that determines whether the recipient wallet is authorized

## Payroll memo fields

Remlo uses a fixed-width payroll memo that carries:

| Field | Why it exists |
| --- | --- |
| `paic` message type | Marks the transfer as payroll-related |
| Employer id | Links the transfer back to the company workspace |
| Employee id | Links the transfer to a specific worker |
| Pay period | Anchors the payout to a payroll cycle |
| Cost center | Supports accounting and departmental segmentation |
| Record hash | Provides a compact integrity link back to the payroll record |

The app builds these values during payroll preparation and exposes decoded memo data in both operator and employee surfaces.

## Where memo data appears

| Surface | Use |
| --- | --- |
| `POST /api/employers/[id]/payroll` | Builds memo values before execution |
| `/portal/payments` | Shows payroll context to employees |
| `/api/mpp/memo/decode` | Returns decoded memo fields for auditors or agents |
| `/api/mpp/payslips/[runId]/[employeeId]` | Includes memo context in a payslip response |

## TIP-403 enforcement

TIP-403 is used in two places:

- inline during payroll preparation so unauthorized recipients are blocked before submission
- through the compliance API so operators, auditors, and agents can check wallets directly

## Compliance event ledger

Compliance outcomes are recorded in `compliance_events`.

| Source | Event examples |
| --- | --- |
| Policy checks | `mpp_check`, `payments_paused`, `payments_resumed` |
| Bridge KYC updates | `kyc_approved`, `kyc_rejected`, `kyc_pending`, `kyc_expired` |

This ledger powers `/dashboard/compliance` and gives Remlo a durable audit trail for both automated and manual review decisions.
