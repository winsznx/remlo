# Bridge Integration

Bridge is the fiat rail inside Remlo. All Bridge calls go through `lib/bridge.ts` or server route handlers. Client components never call Bridge directly.

## Runtime model

| Property | Value |
| --- | --- |
| Production base URL | `https://api.bridge.xyz/v0` |
| Sandbox base URL | `https://api.sandbox.bridge.xyz/v0` |
| Auth header | `Api-Key: <BRIDGE_API_KEY>` |
| Idempotency | `Idempotency-Key` on write requests |
| Verification | `bridge-signature` on incoming webhooks |

## Route mapping

| Remlo route | Bridge operation | Purpose |
| --- | --- | --- |
| `POST /api/employees/[id]/kyc` | `POST /customers`, `POST /kyc_links` | Create employee Bridge customer and return a KYC URL |
| `GET /api/employers/[id]/treasury` | No direct Bridge call on read path | Surface treasury state after deposits are accounted for in Remlo |
| Reserved in master: `POST /api/employers/[id]/treasury` | `POST /customers/{id}/virtual_accounts` | Create a deposit account for employer funding |
| Reserved in master: `POST /api/employees/[id]/card` | `POST /customers/{id}/card_accounts` | Issue a Visa prepaid debit card |
| `POST /api/mpp/bridge/offramp` | `POST /transfers` | Push payroll balance to ACH, SEPA, SPEI, or PIX |
| `POST /api/webhooks/bridge` | Incoming Bridge webhook | Update KYC and transfer state inside Supabase |

## Supported money paths in current code

- Employee KYC links
- Employee Bridge customer creation
- Off-ramp transfers from Bridge wallet balance to bank rails

## Coverage assumptions from the master

The master file assumes Bridge-backed virtual accounts for employer treasury funding and Visa cards for employees in supported regions. The current repo already contains the Bridge client helpers for these paths, but some of the reserved routes are not yet wired into the live Next.js route tree.
